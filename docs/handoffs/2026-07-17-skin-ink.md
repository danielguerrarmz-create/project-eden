## What shipped

`src/scene/skinShader.ts` â€” two `onBeforeCompile` injections into `SurfaceMesh`'s existing
material, mirroring `revealShader.ts`:

- **Contour ink**, a dark line every **0.25 m** of world height, `fwidth`-scaled so it stays a
  constant SCREEN width at any camera distance. This is the direct fix for "illegible from
  above": a smooth vertex-colour gradient reads as nothing from overhead, contour rings read as
  elevation from any angle, because that is what a topographic map is for.
- **Fresnel rim**, grazing-angle darkening, which inverts the eave's liability â€” the edge-on
  condition that makes it vanish is exactly what a Fresnel term detects.

Both mix into `diffuseColor.rgb` â€” albedo, lit by the rig like any other material. Never
emissive, never an overlay. **There is a test asserting exactly that** (`diffuseColor.rgb = mix`
appears twice, `emissive +=` and `gl_FragColor =` appear never), because "anti-Jarvis" is a rule
worth pinning rather than remembering.

**Scoped by construction, not by care:** `SurfaceMesh` is imported and rendered ONLY by

the constraint that bit earlier today is structurally satisfied. Distinct

## SAI'S CHUNK-ORDER CATCH WAS REAL â€” verified, and the failure is worse than predicted

Answering the lead's specific question. Measured against the installed **three@0.169.0**, not
taken on faith in either direction:

```
2267  #include <clipping_planes_fragment>
2505  #include <color_fragment>          <- diffuseColor set here
2692  #include <normal_fragment_begin>   <- `vec3 normal` DECLARED here
2726  #include <normal_fragment_maps>    <- earliest `normal` is final
2879  #include <lights_physical_fragment>
```

**`color_fragment` does run before the normal chunks.** Sai was right, and could not have
checked it.

**The failure mode is louder than the spec said.** Sai wrote that a fresnel there "reads
undefined". It does not: `vec3 normal` is **declared inside** `normal_fragment_begin`, so
referencing it after `color_fragment` is an **undeclared identifier â€” the material fails to
COMPILE**. Not a subtle shading artifact: a dead shader. The split into two injection points is
mandatory, not preferable.

Three more facts that fell out of checking it, all now pinned by tests:

- `normal_fragment_maps` (2726) still precedes `lights_physical_fragment` (2879), so both
  passes land before lighting consumes the albedo. That is what keeps the ink lit.
- `varying vec3 vViewPosition` **is** declared unconditionally in the physical fragment shader,
  and is view-space like `normal`, so Sai's dot product is meaningful.
- **`extensions.derivatives` deliberately OMITTED, against the spec.** The spec's WebGL1
  fallback note is stale for r169: three has been **WebGL2-only since r163**. The build contains
  no `getContext('webgl')` path, emits `#version 300 es`, and the string
  `OES_standard_derivatives` **does not appear anywhere in it**. `material.extensions` defaults
  to `undefined` and nothing reads it. `fwidth` is core in GLSL ES 3.00. Setting it would have
  been dead code.

## Deviations from the spec, and why

- **Constants baked into the GLSL, not 7 uniforms.** `revealShader` uses a uniform because
  `uRevealY` ANIMATES per frame. None of these values ever change, so uniforms would be
  plumbing plus a live class of bug (a uniform object shared, or not shared, across two
  material instances) to buy nothing. They are named TS constants interpolated at module load:
  one place to tune, HMR reloads it, and the values are assertable from a test.
- **Colours converted through `THREE.Color`, and this one is not cosmetic.** ColorManagement is
  on, so `SurfaceMesh`'s vertex colours are LINEAR and the albedo this mixes into is linear.
  Interpolating the naive `0x4a/255 = 0.290` instead of the true linear `0.068` would render an
  ink **four times too light** â€” and would look merely "a bit washed out", which is very hard to
  trace back to a colour-space mistake. Pinned by a test.
- **`max(fwidth(...), 1e-5)`, added.** `smoothstep(a, a, x)` is undefined and `fwidth` is zero
  on any horizontal run â€” which is precisely the CROWN, the flattest and most-looked-at surface
  in the frame. The spec's line would have gone NaN there.
- **`MIN_POLAR` NOT touched.** Sai's follow-on says to relax it from a photographed
  measurement, explicitly not a guess. I cannot photograph. Left at 30Â° for whoever can.
- **P2 wobble NOT built**, per instruction.
