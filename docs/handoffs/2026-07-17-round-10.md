## The one-word fix, and the false claim it exposed (item 8)

**Three of Daniel's four "zoom" notes were ONE bug, and `object-fit` was never involved.**
`FIT_FRAME`'s `max-h-full` resolves against the button; `items-start` left the button's height
`auto`; **a percentage max-height against an indefinite containing block computes to `none`.** The
constraint did not fail, it *evaporated* — the image took its full natural height behind the button's
`overflow-hidden`. **At 1440x760 ALL TWELVE heroes were clipped, worst 47.7% of Synthetic Vision. At
900, three.** `items-start` → `items-stretch`. Verified 0 clipped at both heights and **0 ratio
deviation on every hero**, so the clip was not traded for a distortion.

**CLAUDE.md said "Measured: hero crop 0% across all twelve" and it was false for rounds.** Corrected
in place, because it is the file every session reads first. It was true about object-fit and reached
for the word "crop". **The probe agreed with it, which is why it survived:** `qa/project-media.mjs`
computes crop from the img's rect ratio, and **a clipped `<img>` keeps its natural ratio in
`getBoundingClientRect()`** — the element reports the size it wants to be and the clip happens on an
ancestor's paint. The instrument was never broken; it answered a different question correctly and its
answer got quoted for the question nobody asked. `qa/hero-clip.mjs` compares the IMAGE's rect to the
**button's**, and takes a viewport height, because **the clip depends on region ratio, which rises as
the window shortens. That is why Daniel saw it and a pinned 1440x900 harness structurally could not.**
