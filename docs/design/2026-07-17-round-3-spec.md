# 2026-07-17 — Round 3: the beauty pass (watercolor, vines, joinery, onboarding)

Spec-only, built against `docs/handoffs/2026-07-17-round-3-brief.md` and Senku's research
(`docs/research/2026-07-17-npr-watercolor.md`, whose technical direction is approved and not
re-litigated here: one `InkPass` over normal plus depth, `SKETCH`/`WASH` modes, `MeshToonMaterial`
shading, `VSMShadowMap`). This doc specs the LOOK on top of that direction and covers the other
five brief items in Edward's build order. Everything here is `src/scene/*`, `src/pages/draw/*`,
`src/pages/draw/*`, or `src/ui/*`. `src/engine/*` is not touched.

Governing constraints, repeated because they discipline every item below: `three` chunk stays
under 1100 kB (about 41 kB of headroom today); no em or en dashes in any user-facing copy;
no decorative blue marks; panel register stays serif figures, lowercase mono labels, paper
vellum; explode and bake stay smooth against 200 plus instanced pieces; repo is public.

## Part A: the watercolor and sketch pass (the headline)

### A1. One shared component, mounted in both canvases

New folder `src/scene/npr/`:

- `inkShader.ts`: the fragment shader. One shader, `#define MODE_SKETCH` / `#define MODE_WASH`
  branches, per Senku's recommendation against maintaining two pipelines.
- `InkPass.tsx`: the R3F wiring. `EffectComposer` plus two `RenderPass`es (scene color, and a
  second pass with an override `MeshNormalMaterial` for the normal and depth buffer) plus the
  `ShaderPass` running `inkShader` plus `OutputPass`, all from `three/examples/jsm`, zero new
  dependencies. There is no `@react-three/postprocessing` wrapper installed and Senku's research
  explicitly rejects adding one, so this component owns the render loop override itself: mount it
  with a `useFrame` callback at a priority above zero, which tells R3F to skip its own default
  render for that frame and let the composer render instead. This is a known, documented R3F
  pattern; it does not need a new library.
- `toonGradient.ts`: builds the small `DataTexture` gradient map shared by every `MeshToonMaterial`
  in the scene (timber, steel, foliage), so the banding logic exists in one place, not three.

Mounted once per `Canvas`: in `DrawPage.tsx`'s canvas, after `CinematicCamera` (needs `gl`/`scene`/
`camera` to already exist), and inside `scene/Scene.tsx` for `#/studio`. One `<InkPass mode={...}/>`.

### A2. Sketch pre-bake, watercolor at bake, one shared crossfade

`#/draw` gets both modes; `#/studio` is always resolved, so it only ever needs `WASH`. Concretely:

- `#/draw`: `uMode` starts at `0` (pure sketch) while `!baked`, and crossfades to `1` (pure wash)
  over the same window the bake dissolve already runs. **Reuse `revealProgressRef`**, the ref
  `DrawPage.tsx` already threads into `BakeReveal` for the lattice sweep: `InkPass` reads
  `revealProgressRef.current` in its own `useFrame` and writes it straight into `uMode`. This is
  the same "arrives with the object, not before or after it" timing round 2 used for the count-up
  figure, and it costs zero new state. No new ref, no new clock.
- `#/studio`: `uMode` is a constant `1`. There is no soft phase there to sketch.

### A3. What each mode actually renders

**SKETCH** (pre-bake, `#/draw` only): grayscale. Sobel edges off the normal and depth buffer
(contours and creases, not material or shadow boundaries, per Senku's finding that diffuse-only
Sobel is the wrong signal). A screen-space hatching pass for tonal fill in place of shading. Noise-
driven UV wobble on both the edge taps and the hatch lines for the hand-tremor read. No color, no
Kuwahara. This is the read for the soft, still-editable surface: a maquette, not a rendering.

**WASH** (default everywhere else): color. Same Sobel edges, now composited over a simplified
(non-anisotropic) Kuwahara wash at radius 3 to 4 px, per Senku's recommendation, for the flattened-
into-color-regions brushwork read. Procedural gradient noise for paper grain, sampled low-frequency
and biased into shadow regions (echoing Enscape's "surface detail" control). A desaturation and
shadow color-shift term on top.

### A4. The palette shift: warm, not blue. This is a deliberate deviation from the reference

Both references Daniel gave name a blue shift in the shadows as part of Enscape's signature
watercolor look. **I am overriding that specifically**, and flagging it because Daniel picked
these references and might expect the blue: the house palette's own comment reads "warm paper
base plus charcoal ink, no blue anywhere," and the standing rule is no decorative blue marks even
when the blue would be structural rather than ornamental (this exact tension was flagged and
resolved the same way earlier this session, over the environment map's reflection tint). A shadow
wash that turns the steel or timber blue-grey is the same violation with a fancier justification.
**My call: the wash desaturates and darkens toward a warm umber-charcoal (near `#3a3226`, close
kin to the house's own `ink`/`bark` tones) instead of Enscape's blue.** It still reads as painterly
shadow, it just stays inside the palette the rest of the product already commits to. Flagged in the
open questions below in case Daniel specifically wants the blue anyway, but my working assumption
is no.

### A5. Edge lines, shading, shadows: the three cheap layers, not one expensive one

Per Senku's §7 and §8, split the look across layers rather than doing everything in the post pass:

- **Edge lines**: warm dark ink, near `#2f2a1f` (slightly warmer and lighter than pure `inkBlack`
  so it reads as a drawn line, not a graphic cutout), one weight, Sobel-driven. Threshold and line
  width are eyeball-tunable constants Edward should set against a real render, same discipline as
  every unsourced number in this repo.
- **Shading**: swap `Folly.tsx`'s timber and steel materials from `meshStandardMaterial` to
  `MeshToonMaterial` with the shared 3 to 4 step `toonGradient`, biased toward the warm shadow band
  from A4. `MeshToonMaterial` supports `vertexColors`, so the existing two-tone structural versus
  fastener `instanceColor` split (round 2's "gotcha," the white-buffer-plus-vertexColors recipe)
  carries over unchanged. Also swap `GrowthOverlay`'s (or its replacement's, see Part C) foliage
  material to `MeshToonMaterial` with its own green-leaning gradient, so the living layer paints
  the same way the timber does.
- **Shadows**: `renderer.shadowMap.type = THREE.VSMShadowMap`, set via `onCreated` on both
  canvases (`DrawPage.tsx` currently sets `shadows="soft"`, which resolves to PCF soft; this
  needs the explicit VSM override instead), plus a wider `light.shadow.radius`. Zero bundle cost,
  it is a renderer flag.

### A6. Panel register sits unchanged on top, but check it against a busier backdrop

The DOM overlay panels (`paperVellum/85` with `backdrop-blur`) are already siblings of the canvas,
not inside it, so the ink pass never touches their pixels directly. The one real risk: a grainier,
higher-contrast painterly background behind a semi-opaque panel reads busier than the current flat
render did. **Verify legibility on a real bake; if it suffers, the fix is bumping panel opacity
from `/85` toward `/90` to `/92`, not redesigning the panel.** Same "eyeball it before locking a
number" note as every visual pass in this repo.

## Part C: in-engine plant growth, botanically recognizable per species

### C1. Replace the icosahedron blobs with habit-driven geometry

`GrowthOverlay.tsx`'s architecture is sound and worth keeping: placement follows the strut field's
real density and normal, size eases toward `outputs.growth.coverageFraction`, staggered per-leaf
thresholds give a natural fill, idle sway reads alive. What needs to change is what gets placed at
each cell, driven by the selected species' `habit` (already real data in `engine/types.ts`) rather
than one shape for every plant.

New file `src/pages/draw/speciesVisual.ts`, the same presentation-only pattern as `speciesSwatch.ts`
(hand-authored, flagged as such, tested for "every species has an entry"):

```ts
export interface SpeciesVisual {
  stemForm: 'net' | 'spiral' | 'arch' | 'mat'; // tendril / twining / scrambler / clinging
  petalColor: string; // real-ish bloom color, ties to the swatch per B3
  petalForm: 'star' | 'raceme' | 'bloom' | 'none'; // clematis-star / wisteria-raceme / rose-bloom / ivy has none
  flowerDensity01: number; // how much of the coverage is bloom vs plain leaf
}
```

Four habit families cover all seven species (`engine/types.ts`'s `ClimbingHabit` already has this
shape): **tendril** (clematis, sweet pea) reads as a fine net of thin stem segments with small
star-shaped flower clusters at high-density cells. **Twining** (wisteria, jasmine, honeysuckle)
reads as a spiraled stem wrap (a short helical instanced tube, cheap, one shape reused per cell)
with hanging raceme clusters for wisteria specifically, since its `note` already calls out trunks
thickening to timber scale. **Scrambler** (the rose) reads as arched cane segments with larger,
sparser rose blooms, matching `supportSpacingM: 0.55`'s wide horizontal-support behavior. **Clinging**
(ivy) reads as a dense flat leaf mat with almost no visible flower, matching `stemLoad01: 0.7`'s
near-solid skin and the September to November flowering note that would read as invisible small
green flowers anyway.

Mechanically: keep the existing per-cell placement, threshold, and sway math from `GrowthOverlay`
untouched. Swap the single `icosahedronGeometry` mesh per leaf for two or three small instanced
meshes per species (a stem-segment primitive and, where `flowerDensity01` calls for it, a flower
primitive), so the instance count stays comparable to today's leaf count, just split across two
pools instead of one. This is the same "stay instanced, stay cheap" discipline the joint hardware
already follows.

### C2. Gate it exactly like the species rail

Mount at `{baked && !exploded && !dissolving && <PlantGrowth />}` in `DrawPage.tsx`, same slot logic
as `SpeciesRail`. During explode the members scatter to their own true positions; foliage anchored
to the pre-explode strut field would hang disconnected in space over the wrong pieces, so it hides
for the same reason the species rail already does.

### C3. Growth over time: port the studio's Year 0/1/3 toggle into `#/draw`, do not invent a new control

The store already carries `params.year`, `setYear`, and `outputs.growth` computed off it (proven
in `#/studio`'s `YearToggle`, `App.tsx:145-171`). `#/draw` has zero year concept today. Rather than
inventing a scrub or an auto-advance animation, wire the identical mechanism in: three small pills
reading `GROWTH.years` (`0`, `1`, `3`), reusing `DrawPage.tsx`'s own `Chip` component (already
styled for this exact register) rather than the studio's separate `PANEL`/pill markup, so it stays
visually native to the draw page instead of importing studio chrome. Position: directly under the
species rail on the right edge, same column, small gap. Bake-only, same gating as everything else
in this list.

## Part D: Japanese joinery — near-invisible fasteners, curved members

This reverses round 2's "hardware reads as hardware" ruling. Recorded here as a real reversal, not
a silent one, per the brief's instruction; the next handoff should say the same.

### D1. A genuinely lucky find: the codebase already planned this exact capability

`data/config.ts:103-105` carries a standing `TODO(roadmap)`: *"third system 'timberJoinery', 5-axis
all-timber milled connections (BUGA-style, no visible steel). The node-graph representation the
geometry now emits is exactly what it will consume."* That is item D, already named, already
scoped, already understood as a future real joint system. **This spec does not build that system**
(a real all-timber milled joint changes the solver's cut geometry, which is `src/engine/*`, off
limits, and a much bigger job than a demo round). What it specs is the render-only illusion of it,
sitting on top of the existing `'hub'` solver output, so the demo can show the direction without
the engine claiming a joint family it has not actually engineered yet.

### D2. Hide the hardware behind a flag, do not delete the code path

`Folly.tsx` gains one prop, `hardwareVisible: boolean` (default `false` for this round, `true`
reproduces round 2's exact current behavior). When `false`, the existing `buildSteel` output still
computes (nothing in `connectors.ts` changes), but `Folly` simply does not mount the box and
cylinder instanced meshes for the fastener-role instances (bolts, the round 2 dome-nut caps, the
ground-screw stubs). Structural-role plates and drums (the hub core disc, ring flanges, base
plates) are handled differently, see D3. This is a two-line conditional at the mount site, and
because `SteelRole` already exists on every cylinder instance from round 2, no new tagging is
needed to know which is which.

### D3. The gap the hidden hardware leaves has to be closed with something real, not left empty

The solver's `hubStandoffM` (`jointGeometry.ts:283-328`) computes a real clearance so timber ends
stop short of the connector envelope, sized off `JOINTS.hub.coreDiaMm` (140 mm) plus
`envelopeClearanceMm` (10 mm), both real named constants. That gap exists whether or not steel is
drawn there. Simply hiding the steel leaves a floating disconnect between timber ends, which reads
as broken, not concealed. **Fix: render a timber-toned collar over that same envelope**, sized to
the same `coreDiaMm` plus clearance the engine already computes, oriented on the node normal,
material-matched to whichever stock is arriving there (C24 tone for lattice, LVL tone for ring).
This is the identical "only render a part the config already names" discipline round 2 held itself
to for the dome nut and ground screw: the Ø140 mm envelope is a real, already-computed dimension,
this only changes what covers it. Visually this reads as a wrapped timber sleeve concealing a
hidden joint, the honest architectural cousin of a scarf joint, not a literal through-tenon with
visible peg ends (which would require inventing peg proportions with no spec line behind them,
exactly the trap round 2's own header warns against). At lamella nodes, where the joint is a single
visible through-bolt rather than a hub cluster, the equivalent move is smaller: replace the round 2
dome-nut cap with a small timber plug cap over the bolt head, same size class as `DOME_NUT_DIA_M`,
timber-toned instead of steel-toned.

### D4. Curved members: fix the ring facet before inventing a stylistic bend

`memberPrism` extrudes every member as a straight prism between two cut planes. For the diagrid
struts (`lattice`, `foot` in the hub system) that is honest: `STOCK.strut` is planed C24, genuinely
straight sawn stock, and curving it would misrepresent the material the same way an invented hub
boss would misrepresent the joint. **Struts stay straight. Do not curve them.**

The eave and crown rings are a different case, and a real one: `STOCK.blank` pieces are already
described in this codebase as "curved pieces cut from sheet stock" (`GRAMMAR.eaveBlanksPerFootSpan`
comment), and `jointGeometry.ts`'s own mitre comment refers to "the faceted representation of a
curved piece." **The ring is a physically curved beam that the current renderer draws as a straight
chord between adjacent nodes**, which is exactly why it reads as a polygon of straight sticks
rather than a bent glulam ring. Fixing that is not a stylistic invention, it is correcting a render
approximation toward the fabrication truth the comments already state. Concretely: subdivide each
ring member's prism into 3 to 5 longitudinal segments instead of one, and bow the intermediate
cross-sections outward from the straight chord toward the TRUE ellipse point at that same
parameter, using the plan center and `planA`/`planB` (already exposed on `outputs.geometry`, read
today by `PlacedScaleFigure`/`entryBearing.ts`) rather than an invented bow fraction. If each ring
node does not already carry its own angular position on the plan ellipse, deriving it once via
`atan2` around the plan center is cheap and only needs to happen at render time in `Folly.tsx`, not
in the engine.

Triangle cost, worked the same way round 2 worked the chamfer: a chamfered ring member today is 28
tri (2 caps at 6 tri plus 8 sides at 2 tri). At 4 segments, the two end caps stay fixed (segment
joins are shared, not new caps) but the side faces multiply by segment count: 8 sides times 2 tri
times 4 segments equals 64, plus the 12 cap tri, **76 tri per ring member**, about 2.7 times round
2's per-member figure. This only applies to ring members, a small subset of the whole kit: a
typical 3 to 4 foot design has on the order of 12 to 16 eave and crown members total against 150
plus lattice struts staying at 28 tri each, so the absolute delta is modest even though the
per-member multiplier looks large. **Someone should confirm on a real GPU frame-timing capture
before calling it free, same caveat every prior handoff here has carried.**

Lamella pieces (the second joint system, not the demo default but present in the codebase) get an
optional, smaller secondary bow, purely an art-direction touch, not fabrication-sourced the way the
ring fix is: flagged explicitly as my own proportion, lower priority, build only if the ring fix
and the rest of this round leave headroom.

## Part E: the scale figure, options and a recommendation

`ScaleFigure.tsx`/`PlacedScaleFigure.tsx` are unusually well-reasoned already: static, faceless,
billboarded, placed once at a solved entry bearing, with an explicit "the kitsch line is the entire
risk" header. Any option here has to respect that the figure itself staying still, faceless, and
un-costumed is not the thing to fix; Daniel's ask was "displayed better or a more interesting
interaction," not "make it more alive."

**Option A, recommended: a dimension callout, not a new figure behavior.** Add a slim line (the
same graphic language architectural drawings already use, and this exact idiom already exists in
this codebase's own mono-labeled readouts) connecting the figure's crown to the nearest eave or
crown datum, with the real height in meters appearing beside it in the house mono register.
Triggered the same way the commission figure "arrives with the object": briefly on bake settle,
then available again on hover or click if a viewer wants to re-check it. This turns the figure from
a passive size comparison into an active measurement device without adding any motion, clothing, or
personality to the figure itself, the exact places round 1's kitsch-avoidance rules drew the line.
Cheapest of the three, ships this round, on-brand by construction since it reuses vocabulary the
product already speaks.

**Option B, not recommended, name it to rule it out:** multiple figures at different vantage
points as the camera orbits. `entryBearing.ts`'s own header explicitly rejected a figure that
slides to follow the camera as "the most obviously fake thing in the shot," and solving that
problem for a SECOND figure compounds the exact failure mode it was written to avoid, plus adds
clutter risk on a filmed frame. Reject.

**Option C, a stretch, not this round's scope: let the viewer place the figure.** Reuse the same
pointer-to-ground raycast `DrawStage` already does for draw and sculpt, gated to bake-only, so a
viewer (or Daniel, filming) can click a spot on the apron to check scale exactly where they want
it, replacing the automatic entry-bearing placement on demand rather than instead of it. This is
the genuinely "more interesting interaction" answer, and it is real, buildable work: a new pointer
mode, a small draggable ground marker, and a decision about what happens to the automatic
entry-bearing placement when a manual one exists. Flagged as a v2 candidate if there is time after
the rest of this round, not a build-order item today.

**My recommendation: ship A now, hold C as a stretch if time allows, do not build B.**

## Part F: guided onboarding

### F1. What's actually missing, precisely

Today's guidance rail (`RAIL_LINES`, `toolCopy.ts:46`) already teaches turning and zooming, but two
real gaps remain. First, timing: it only appears "once the first canopy stands" (per this session's
own prior guidance-and-skin spec), which means the very first few seconds, exactly when someone is
deciding how to even approach the lawn, teach nothing about navigation at all. Second, content: the
brief asks for left versus right click to be taught explicitly, and the rail's current two lines
only state the orbit gesture, not what a plain left-drag does. **Also: the brief asks for pan to be
taught. `DrawPage.tsx`'s `OrbitControls` has `enablePan={false}`.** There is no pan in this canvas.
Teaching a gesture that does not exist would be the kind of overclaim this codebase's own honesty
discipline exists to catch; the fix is not adding pan, it is not promising it in the copy.

### F2. The fix: earlier timing, one more line, no new UI mechanism

Two small, surgical changes to the existing rail, not a new tutorial component:

1. **Move the rail's mount trigger from "first canopy stands" to component mount.** The rail
   should be the very first thing on screen, before a single line is drawn, not after. Everything
   else about it (the 6.5 s auto-hide, the dismiss-on-first-orbit-or-drag, the reset on start over)
   stays exactly as already built; only the trigger condition changes.
2. **Rewrite the two lines to cover both clicks honestly:**
   `'left-drag uses the tool · right-drag, or hold space, turns it'` replaces the current first
   line (states what left-drag actually does, generically enough to stay true across draw, sculpt,
   and excavate, without promising pan). `'scroll to zoom'` stays as the second line, unchanged.

### F3. "Skippable" is already built, name the mechanism instead of adding a button

The rail already dismisses the instant the user orbits (`onStart` on `OrbitControls`), and that is
the same "guidance that has been obeyed should stop talking" rule already documented at the site.
**That dismiss-on-first-interaction behavior already satisfies "skippable."** Adding an explicit
"skip" button would be new chrome solving a problem the existing mechanism already solves, and
would contradict the minimal-UI direction the rest of this codebase holds to. No new control.

### F4. Re-armable for takes: already true, verify it stayed true

The existing rail resets its shown state on "start over" (a bug fix from earlier this session).
Nothing in F1 through F3 touches that reset; Edward should just confirm it still fires correctly
once the trigger condition changes from "first canopy" to "mount," since the reset and the mount
trigger are now doing more overlapping work than before.

## Non-goals, explicit

- **No `@react-three/postprocessing` or `postprocessing` package.** Senku's research rejected it
  as a duplicate of what `three/examples/jsm` already provides; nothing in this spec needs it.
- **No `OutlinePass`, no drei `<Outlines>` as the primary edge technique.** Both are wrong for 200
  plus animated `InstancedMesh` pieces per Senku's research; the Sobel-on-G-buffer approach is the
  one built here.
- **No literal through-tenon pegs, no invented hardware silhouette.** The joinery concealment in
  Part D covers a real, already-computed envelope dimension; it does not invent a new shape with no
  spec line behind it.
- **No curved struts.** Straight C24 stock stays straight; only the genuinely curved-cut ring
  pieces get the bow, and it is derived from real plan geometry, not a stylistic percentage.
- **No pan gesture, taught or added.** `enablePan={false}` stays; the onboarding copy stops
  implying otherwise instead of the control being changed.
- **No second scale figure, no figure that slides to track the camera** (Option B, explicitly
  rejected in Part E).
- **No new "skip" button for onboarding.** The existing dismiss-on-interact mechanism already is
  the skip.
- **No blue shadow wash**, despite it being part of both Enscape references. See A4.
